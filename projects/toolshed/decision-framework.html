Layer 1: Pre-Build Intelligence designing

Decision Framework

Structured decision trees for go/no-go, scope, and positioning. Turns fuzzy ideas into clear build-or-kill decisions.

The problem

Most failed builds fail before code starts. The idea was wrong, the market was wrong, or the scope was wrong. But we usually don't discover this until we've invested significant build time.

Ad-hoc thinking doesn't catch these issues reliably. You need structured questions that force clarity — and a process that makes "no" a valid output.

The approach

An MCP server that walks through a decision tree for any new idea:

Problem validation Is this a real problem? Who has it? How do they solve it today?
Market assessment Is this a market I can reach? Is it big enough? Is it growing?
Competitive landscape Who else is solving this? What's my angle?
Build feasibility Can I build this in a week? What's the minimum testable version?
Signal clarity What would success look like? How would I know to kill it?
Output

A structured decision document with:

  • Go/no-go recommendation with reasoning
  • Key risks and unknowns
  • Suggested scope for v1
  • Success/failure criteria

Human makes the final call. The framework provides structure, not answers.

Status

Current: Designing the decision tree structure. Running it manually on each new idea to validate the questions.

Next: Formalize as MCP server once the manual process stabilizes.
