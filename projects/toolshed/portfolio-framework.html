Portfolio Framework

Dashboard for tracking products, signals, and graduation decisions. Kill/continue/pivot structured as a process.

The problem

Shipping is the beginning, not the end. After launch, you need to track signals, make kill/continue decisions, and manage a portfolio of experiments — without getting emotionally attached to any single build.

Most solo builders either kill too early (no patience for signal) or too late (sunk cost fallacy). A structured framework helps make these decisions objectively.

The approach

An MCP server for post-launch portfolio management:

Signal tracking Revenue, engagement, and custom metrics defined at build time.
Time-boxed evaluation Forced decision points — no indefinite "let's see how it goes."
Decision framework Structured criteria for kill/continue/pivot/graduate.
Portfolio view All experiments in one place with status and next actions.
Decision options

At each evaluation point, four possible outcomes:

◆ Kill — No signal, no path to signal. Archive and capture learnings.
◆ Continue — Signal is present but not conclusive. Extend evaluation period.
◆ Pivot — Signal suggests different direction. Scope new experiment.
◆ Graduate — Strong signal. Move to growth program for dedicated investment.
